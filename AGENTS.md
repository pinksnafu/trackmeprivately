## 🏗️ Architecture & Folder Structure

This application is built as a single, unified codebase using **Next.js** (App Router). It serves both the telemetry ingestion API and the secure admin dashboard.

```text
/
├── app/                  # Next.js App Router Pages and API routes
│   ├── api/
│   │   ├── auth/         # WebAuthn/Passkey registration and login flows
│   │   └── collect/      # Telemetry ingestion endpoint (tracker hits)
│   ├── login/            # Secure biometric login page
│   ├── globals.css       # Premium Dark-Mode theme (Vanilla CSS only)
│   └── page.tsx          # Analytics dashboard Server Component
├── components/           # Reusable UI Components
│   ├── Chart.tsx         # Area chart (Client component, uses Recharts)
│   └── WebsiteSwitcher.tsx# Website switcher dropdown (Client component)
├── lib/                  # Shared library modules
│   ├── auth.ts           # JWT session crypt/decrypt helpers (using jose)
│   ├── prisma.ts         # Singleton database Prisma Client
│   └── setup.ts          # Bootstrapping setup token manager
├── prisma/               # Database migrations and SQLite schema
│   ├── schema.prisma     # Active Prisma schema models
│   └── dev.db            # Default SQLite local database
├── public/
│   └── tracker.js        # The client-side zero-cookie tracker script
├── proxy.ts              # Edge Proxy (Middleware) route guard
└── package.json          # Node dependencies and scripts
```

## 🗄️ Database Models (`schema.prisma`)

*   **`User`**: Admin profile containing the unique username.
*   **`Authenticator`**: Holds the public keys, counter, and transports for FIDO2/WebAuthn Passkeys bound to the user.
*   **`Website`**: The domains configured to track analytics for.
*   **`Event`**: Individual pageview or custom action telemetry events.
    *   `sessionId` is generated daily via `SHA256(IP + UserAgent + WebsiteID + DateSalt)` to anonymize visitors completely.

## 🔒 Authentication Flow (WebAuthn & Passkeys)

Instead of passwords or email servers, the admin dashboard is locked down via biometric Passkeys:
1.  **Stateless Challenges**: Challenges generated by `@simplewebauthn/server` are stored in an encrypted, short-lived HTTP-only cookie (`auth_challenge`) using `jose` rather than server memory or database states.
2.  **Bootstrap Security (Auto-Log Token)**:
    *   If no `User` exists in the database, the server generates a random 8-character token on boot, logs it to console, and writes it to `prisma/setup_token.txt`.
    *   The setup UI requires this token to associate the first Passkey as the admin owner.
    *   Once a user is registered, the file is deleted, and the path is permanently closed.
3.  **Route Protection**: Next.js Proxy/Middleware (`proxy.ts`) checks for a signed JWT (`session_token`) cookie on all dashboard routes.

## 📡 Key API Endpoints

### 1. Ingestion: `POST /api/collect`
*   Accessible from any origin (CORS allowed).
*   Expects: `{ event, url, referrer, width, website }`.
*   Parses incoming user-agent parameters, maps screen resolutions, creates the daily session hash, and inserts the telemetry record asynchronously.

### 2. Registration Options: `POST /api/auth/register/options`
*   Expects: `{ username, token }`. Validates the setup token.

## 🛠️ Standards & Rules for AI Coding Agents

If you are an AI assistant writing code for this repository, you **MUST** adhere to the following:
1.  **No Cookies for Ingest**: The client-side tracker (`public/tracker.js`) must remain pure vanilla JavaScript, contain zero external dependencies, and must **never** read, write, or access cookies, `localStorage`, or `sessionStorage`.
2.  **Vanilla CSS Only**: Do not install or import utility CSS frameworks like Tailwind CSS. All styling tokens are maintained globally in `app/globals.css` using custom properties.
3.  **NEXTAUTH_SECRET in Production**: The application must enforce that `process.env.NEXTAUTH_SECRET` is set when running in production (`process.env.NODE_ENV === 'production'`). Throw an error on boot if missing.
4.  **Edge Proxy Convention**: Do not rename `proxy.ts` back to `middleware.ts` as Next.js 16 deprecated `middleware.ts` in favor of proxy files. Additionally, the proxy file MUST export either a default function or a function named `proxy` (do not export a function named `middleware`).
5.  **Strict WebAuthn Domain Checks**: Allow overriding WebAuthn domain validation in production via `process.env.ALLOWED_RP_ID` and `process.env.ALLOWED_RP_ORIGIN` to prevent DNS/Host spoofing.
6.  **Build-Time Environment Validation**: Avoid throwing strict environment configuration errors (e.g. missing `NEXTAUTH_SECRET`) at the module-evaluation level (top-level scope) of files imported by routes. This will cause `next build` to crash. Always check and throw configuration errors inside runtime execution functions.
